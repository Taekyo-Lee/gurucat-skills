# API Reference

Usage reference for skill-updater commands.

---

## `/skill-updater --check`

Scan all installed skills and report which ones have upstream updates available.
This is a **read-only** command — it checks status but changes nothing.

### Usage

```
/skill-updater --check
```

### What it does

1. **Scans all skill locations** — both user-scope (`~/.claude/skills/`,
   `~/.opencode/skills/`, etc.) and project-scope (`./.claude/skills/`, etc.).
   Any folder containing a `SKILL.md` with YAML frontmatter is recognized as
   a skill.

2. **Classifies each skill's origin** — reads the frontmatter to determine
   whether the skill has `metadata.source` (knows where it came from) or not
   (local/unknown).

3. **Fetches the latest repo tag** — for skills with `metadata.source`, checks
   the upstream repo for the newest version tag. Skills from the same repo are
   grouped so only one tag lookup is needed per repo, not per skill.

4. **Diffs each skill's files between tags** — a new repo tag doesn't mean
   every skill changed. The check compares the skill's specific files between
   the user's `repo_tag` and the latest tag to determine if that skill actually
   has updates.

5. **Displays a status table** — shows all found skills with their version,
   repo tag, and update status.

### Output

```
Skill                      Location                  Version          Repo Tag            Status
─────────────────────────  ────────────────────────  ───────────────  ──────────────────  ──────────────────
code-review                ~/.claude/skills/         1.0.0 → 1.2.0   v2.3.0 → v2.5.0    Update available
deploy                     ./.claude/skills/         2.0.1            v2.5.0              Up to date
my-helper                  ~/.claude/skills/         0.5.0            —                   Local (no upstream)
shared-skill               ~/.claude/skills/         —                —                   Unknown source
```

### Column descriptions

| Column | Description |
|---|---|
| **Skill** | The skill's folder name |
| **Location** | Where the skill was found (user-scope or project-scope) |
| **Version** | Per-skill `metadata.version`. Shows `old → new` if an update is available, or just the current value if up to date |
| **Repo Tag** | Repo-wide `metadata.source.repo_tag`. Shows `old → new` if the repo has a newer tag, or just the current value if up to date |
| **Status** | One of the status values below |

### Status values

| Status | Meaning |
|---|---|
| **Update available** | The skill's files changed between the user's repo tag and the latest. Run `/skill-updater <name>` to update. |
| **Up to date** | Either the repo tag is already the latest, or the repo has a newer tag but this skill's files didn't change between them. |
| **Local (no upstream)** | The skill has no `metadata.source` — it was written locally or generated by AI. Nothing to check. |
| **Unknown source** | The skill has no `metadata.source` and its origin is unclear. Run `/skill-updater <name>` to identify and stamp it. |
| **Non-GitHub source** | The skill's `metadata.source.url` points to a non-GitHub host. Automatic checking is not supported; check manually. |
| **Check failed: \<reason\>** | The check couldn't complete for this skill (e.g., auth required, network error, repo not found). The reason is shown. Other skills are still checked. |

### Key behaviors

- **One failure doesn't block the rest.** If a fetch fails for one repo, that
  skill is marked as "Check failed" and the scan continues.
- **Repo-level grouping.** Multiple skills from the same repo share a single
  tag lookup, avoiding redundant API calls.
- **No file changes.** `--check` never modifies any files, creates backups,
  or writes to disk. It is safe to run at any time.

### Examples

Check all skills:
```
/skill-updater --check
```

Typical follow-up after seeing "Update available":
```
/skill-updater code-review
```
